
- `g4l3r1a-3d-jwt-s3cr3t-2026-d0-pr0d` — JWT secret backend
- `DO8013L487N2AL3FB4ND` — DigitalOcean Spaces Access Key
- `/OuRpuWy290o9s3ldGaZLZP2PGgMh+0gutpdqsyyI7I` — DigitalOcean Spaces Secret Key

### Vector de ataque hipotético

Con repo público:
1. Bots/crawlers como GitGuardian/Trufflehog indexan repos públicos en minutos
2. DB password → conexión directa a PostgreSQL prod (puerto 5432 podría estar expuesto o accesible vía SSH tunneling)
3. JWT secret → falsificación de tokens válidos sin necesidad de password
4. Spaces keys → upload/delete arbitrario en bucket prod (vandalism, hosting de contenido malicioso)
5. Admin pass → login completo en `https://ceopacademia.org` con todos los privilegios

### Fase 2 — Rotación completa de credenciales

Todas asumidas comprometidas, rotadas:

|---|---|---|

### Fase 3 — Sanitización del código

Commit `0a55891` — sanitize credentials post-rotation incident:

- `backend/set-cors.js`: refactor a leer `process.env` (vía `dotenv`) — credenciales ya no hardcoded en código ejecutable
- `docs/deploy.md`, `.claude/skills/*/SKILL.md`: valores reales → placeholders `<ver iCloud Keychain o .env del droplet>` con nota de seguridad
- `docs/session-logs/2026-04-13.md`, `2026-04-14.md`: referencias históricas sanitizadas (marcadas como "rotado 2026-04-30")
- `.gitignore`: añadido `.claude/settings.local.json` (per-developer, contenía DB password en args de comandos autorizados)
- `git rm --cached .claude/settings.local.json` (archivo sigue local, fuera del repo)

Verificación `grep` final por las 5 credenciales viejas en archivos rastreados → **cero matches**.

**Decisión consciente:** no reescribir historial git (BFG Repo-Cleaner). Razón: las credenciales viejas ya no funcionan tras rotación, su valor ofensivo es nulo. Reescribir historial implica `--force push` destructivo que rompería el clone del colaborador. Costo/beneficio no lo justifica.

### Fase 5 — Hardening GitHub

Commit `a4667c7` — CODEOWNERS + SECURITY.md:

- **`.github/CODEOWNERS`** — paths sensibles (`backend/`, `migrations/`, `.github/`, `.claude/`, `docs/deploy.md`, lockfiles) requieren review explícito del owner en PRs
- **`.github/SECURITY.md`** — política para colaboradores: nunca commitear secrets, flujo de branches, 2FA, reportes de vulnerabilidades, historial del incidente

Configuración via GitHub web (Carlos):

- **Branch protection en `main`**:
  - Require PR before merging + 1 approval mínimo
  - Dismiss stale approvals when new commits pushed
  - Require review from Code Owners
  - Require status checks + branches up-to-date
  - Require conversation resolution
  - Require linear history
  - **Do not allow bypassing the above settings** ⭐
  - Force pushes OFF, deletions OFF
- **Dependabot habilitado** (gratis): alerts + security updates + grouped security updates + malware alerts
- **Secret Scanning + Push Protection**: NO disponible (requiere GitHub Advanced Security, pago en repos privados de cuentas personales). Alternativa: `gitleaks` local (agendado como tarea futura)
- **Colaborador `creaciondigital-ueb`**: rol "Collaborator" (default en repos personales — sin roles granulares disponibles). La defensa pasa de "rol limitado" a "branch protection + obligación de PR + CODEOWNERS"

## Decisiones de seguridad documentadas

1. **No reescribir historial git** — credenciales rotadas son inertes; el costo del `--force push` no se justifica
2. **No usar PAT permanente para Claude** — discutido honestamente con Carlos: tokens efímeros con scope mínimo y expiración por tarea son mejor práctica que acceso permanente para un agente
3. **Mantener Colaborator (no roles granulares)** — los repos personales de GitHub no ofrecen roles granulares. Migrar a Organization queda como opción futura cuando el equipo crezca
4. **Secret Scanning con `gitleaks` local en lugar de GitHub Advanced Security** — gratis, defensa-en-profundidad. Implementación en sesión aparte

## Lecciones aprendidas

1. **Auditar el repo público al inicio de cada proyecto** — el repo arrancó público sin que el owner notara. Verificar visibilidad debió ser parte del setup inicial.
2. **Las credenciales NUNCA deben commitearse** ni siquiera en docs/skills internos. Usar placeholders + referencia a `.env` o password manager desde el inicio.
3. **`.claude/settings.local.json` debe estar en `.gitignore` por defecto** — su propósito es local, contiene comandos autorizados que pueden incluir credenciales en args inline.
4. **`pg_dump` + scp en sesiones de trabajo deja credenciales en logs y bash history** — futuras sesiones usar variables de entorno o leer del `.env` directamente.

## Pendientes / Próximas vías

- **Gitleaks** (pre-commit hook local + GitHub Action en PRs) — agendado como tarea futura, no bloqueante
- **Revisar accesos SSH** al droplet — Claude tiene SSH permanente; evaluar si rotar SSH keys periódicamente
- **Migrar el repo a GitHub Organization** cuando el equipo crezca a 3+ personas — habilita roles granulares (Read/Triage/Write/Maintain/Admin)
- **Notificar al colaborador**: avisarle del rename del repo (`git remote set-url`), pedirle leer `SECURITY.md`, confirmar 2FA en su cuenta
- **Revocar el DO API Token** generado para esta sesión (scope `spaces_key:*`, 30 días) — recordatorio pendiente al cierre

## Segunda parte de la sesión — Release v3.3.1

Tras cerrar el incidente, Dependabot reportó **3 vulnerabilidades** en dependencias que detectó al activarse durante el hardening:

| Severidad | Paquete | Origen |
|---|---|---|
| Moderate | `postcss <8.5.10` | Frontend build tool |
| **High** | `fast-xml-builder` | Transitive de `@aws-sdk/client-s3` (DO Spaces) |
| Moderate | `fast-xml-parser <5.7.0` | Transitive del mismo |

**Impacto real:** bajo (postcss procesa solo CSS interno, fast-xml-* solo parsea respuestas de DO Spaces controladas), pero mejor práctica = no quedarse con CVEs conocidos.

### Acciones ejecutadas

1. **`npm audit fix`** en frontend + backend → 0 vulnerabilities en local
2. **Bump `package.json` 3.3.0 → 3.3.1** + CHANGELOG entrada `[3.3.1]`
3. **Commit `6042938`** — `chore(release): v3.3.1 — fix 3 Dependabot vulnerabilities`
4. **Deploy backend al droplet**: backup `.env` + `scp package*.json` + `npm ci` en `/var/www/galeria-api/` + `pm2 restart`. Verificación: `0 vulnerabilities` también en prod
5. **Smoke prod**: `/api/health` OK, frontend HTTP 200
6. **Git workflow inicial bloqueado por branch protection** — primer test del hardening configurado funcionó: `git push origin main` rechazado con:
   ```
   - This branch must not contain merge commits. (Linear history)
   - Changes must be made through a pull request.
   ```
7. **Ajuste de branch protection** — Carlos descartó "Require approvals: 1" + "Do not allow bypassing" tras discusión honesta: las políticas estaban dimensionadas a equipo grande, no a proyecto educativo con un owner. Se conservó la protección real (force push, deletions, linear history, require PR) pero se permitió bypass de admin para que Carlos pueda hacer releases legítimos sin clicks extras en GitHub web
8. **Push final exitoso**: `git merge --ff-only` (fast-forward) + tag `v3.3.1` + push main + tags. GitHub registró `Bypassed rule violations` (auditado, válido)
9. **Vuelta a `develop` + sync** (regla 2.6 cumplida)

### Lección sobre branch protection

> *"Las políticas son para protegerte, no para complicarte. Configurar protection enterprise-grade para un proyecto educativo con un owner termina poniendo más trabajo manual que defensa real. La protección correcta para este contexto: force push off, deletions off, linear history, require PR — sin require approval ni bloqueo de bypass admin."*
